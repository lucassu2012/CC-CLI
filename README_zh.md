# IOE - 智能运营引擎（Intelligent Operations Engine）

**面向通信运营商网络运维与市场运营的AI原生Agent Harness平台**

IOE是一款轻量级、可叠加的Agent Harness平台，能够赋予现有电信系统（SmartCare、AUTIN等）AI驱动的智能推理能力——无需替换原有系统。基于Claude Code Agent Harness架构的实战验证模式构建，IOE将"模型是CPU，Harness是操作系统"的理念应用于电信领域。

> "最好的Agent产品，是由理解自己的工作是Harness而非Intelligence的工程师构建的。"

## 核心设计理念

IOE遵循从Claude Code及行业最佳实践中提炼的七大Harness工程原则：

| # | 原则 | IOE实现方式 |
|---|------|-----------|
| 1 | **上下文是最稀缺的资源** | 多策略压缩引擎（Auto/Snip/Collapse/Micro），92%阈值触发 |
| 2 | **原语优于集成** | ~18个可组合的电信工具原语，而非数百个专用连接器 |
| 3 | **收缩的Harness** | 所有脚手架均设Feature Flag；随模型能力提升而逐步移除 |
| 4 | **分离式自我评估** | Generator/Evaluator模式——不同Agent负责创建和评估 |
| 5 | **安全优先的权限** | 拒绝优先的L1-L5信任层级，配合数字孪生预验证 |
| 6 | **非确定性LLM的确定性Hook** | 21个生命周期事件，4种处理器类型 |
| 7 | **为删除而设计** | 每个决策树、工作流、规则都有过期检查点 |

## 核心特性

### 1. TAOR循环引擎
驱动所有Agent行为的核心"笨循环"：
```
思考（Think）→ 执行（Act）→ 观察（Observe）→ 重复（Repeat）
```
- 模型驱动决策（无硬编码工作流约束）
- 流式工具执行（模型输出的同时即开始执行工具）
- 只读工具最多10个并行执行，写入工具串行执行
- 多策略错误恢复，指数退避重试

### 2. 五大领域Agent + 16个子Agent

**规划Agent** — 商网协同的网络规划
- 价值洞察子Agent：商业机会发现、流量预测
- 网络仿真子Agent：覆盖/容量/性能仿真
- 市场收益预测子Agent：网络投资带来的市场收益
- 收益预估子Agent：综合投资回报分析

**网络优化Agent** — 全网自动化优化
- 实时优化子Agent：实时参数调优，追求KPI最优
- 工程优化子Agent：新站/新小区参数优化
- 事件保障子Agent：突发话务处理（音乐会、体育赛事等）

**体验保障Agent** — 主动式用户体验管理
- 投诉预警子Agent：预防性问题发现与主动关怀
- 差异化体验子Agent：5QI/调度优先级管理
- 确定性体验子Agent：面向Top 5%用户的网随人动

**网络运维Agent** — 主动式网络维护与故障管理
- 运维监控子Agent：实时监控与自动修复
- 故障分析子Agent：跨域根因分析
- 上站维护子Agent：现场维修指导

**运营支撑Agent** — 智能化营销决策支持
- 潜客识别子Agent：精准客户定位
- 实时营销子Agent：需求时刻精准触达
- 离网维挽子Agent：预测性客户挽留

### 3. NETWORK.md标准
电信版的CLAUDE.md——声明式、工程师可编辑的知识文件，定义：
- 网络拓扑与域边界
- SLA要求与KPI阈值
- 升级策略与路径
- 禁止操作（如"高峰期永不重启核心网元"）
- 维护窗口与变更冻结期
- 厂商特定注意事项
- 监管合规要求

### 4. L1-L5权限模型（集成数字孪生验证）

| 级别 | 操作类型 | 审批要求 |
|------|---------|---------|
| L1 只读 | 查询KPI、查看告警、读取配置 | 始终允许 |
| L2 低风险写 | 创建工单、安全范围内调参 | 自动审批 + 日志 |
| L3 中风险 | 配置变更、流量重路由 | 需人工审批 |
| L4 高风险 | 设备重启、影响业务的变更 | 高级审批 + 数字孪生仿真 |
| L5 紧急 | 关键故障旁路模式 | 强制事后审计 |

### 5. 上下文压缩引擎
四大压缩策略管理PB级电信遥测数据：
- **AutoCompact**：92%上下文窗口触发，6.8倍压缩比，语义损失<3%
- **SnipCompact**：截断边界外旧消息，保留助手的"受保护尾部"
- **ContextCollapse**：懒提交坍缩，仅在413错误时激活
- **MicroCompression**：持续后台运行——基于时间清除过期结果，基于大小截断

### 6. 六层记忆体系 + Dream整合

| 层级 | 内容 |
|------|------|
| L1 托管层 | 企业/MDM策略 |
| L2 项目层 | 项目根目录的NETWORK.md |
| L3 规则层 | 领域特定规则 |
| L4 用户层 | 运维人员偏好 |
| L5 本地层 | 按目录自动发现 |
| L6 Dream层 | Dream系统自动生成 |

Dream整合作为后台进程运行：
定位（Orient）→ 收集（Gather）→ 整合（Consolidate）→ 修剪（Prune），保持索引在200行/~25KB以下

### 7. 自学习知识库
每次任务闭环自动生成知识条目：
- 故障记录（含根因和解决方案）
- 已验证的修复流程
- 经验教训
- 周期性整合：合并重复项、修剪过期条目

### 8. 数字孪生预验证
所有L3+操作必须通过数字孪生仿真验证：
- 覆盖、容量、性能、体验仿真
- 配置变更影响预测
- 故障注入弹性测试
- 风险评估（高危自动中止）

### 9. Hook系统（21个生命周期事件）
围绕非确定性LLM推理提供确定性保障：

| Hook类型 | 电信应用场景 |
|----------|------------|
| 操作前Hook | 对照合规规则验证配置；检查变更窗口 |
| 操作后Hook | 记录审计追踪；触发监控 |
| 安全Hook | 写操作前在数字孪生上仿真 |
| 合规Hook | 确保符合当地电信法规 |

### 10. 多协议数据集成
通过标准协议连接传统系统：
Kafka、gRPC、REST、SNMP、NETCONF、数据库

支持实时（亚秒级）和非实时数据接入。

## 配置

IOE采用级联配置系统：
```
企业策略 → 运营商配置 → 项目NETWORK.md → 用户偏好 → 默认值
```

通过环境变量配置：
```bash
IOE_OPERATOR_NAME="中国移动"
IOE_REGION="guangdong"
IOE_MODEL_ID="pangu-telecom-72b"
IOE_CHAT_PORT=8080
```

或通过代码配置：
```typescript
import { IOEApplication } from '@ioe/intelligent-operations-engine';

const app = new IOEApplication({
  operatorName: '中国移动',
  region: 'guangdong',
  defaultModelId: 'pangu-telecom-72b',
  digitalTwinEnabled: true,
  separatedEvaluation: true,
});

await app.initialize();
await app.startCli();
```

## 与现有系统的关系

```
IOE 不替换现有系统，而是叠加在现有系统之上。

┌─────────────────────────┐
│         IOE             │  ← AI推理层（新增）
│   Agent Harness         │
├─────────────────────────┤
│  SmartCare │ AUTIN │ADN │  ← 现有产品（不变）
│  DataCube  │ 900+APP│DT │
├─────────────────────────┤
│  网络设备               │  ← 物理/虚拟网络
│  eNB │ gNB │ Core │ TN  │
└─────────────────────────┘
```

- **SmartCare**：提供DataCube（200+主题域）、体验分析 → IOE通过共享数据模块访问
- **AUTIN**：提供900+ APP、故障管理、维护工作流 → IOE通过OSS连接器访问
- **DTN**：提供数字孪生仿真环境 → IOE用于操作前预验证

## 面向未来的设计

遵循"收缩的Harness"原则，IOE的脚手架被设计为可渐进移除：

```typescript
featureFlags: {
  'use_decision_tree_fallback': true,    // 当模型能处理所有场景时移除
  'enforce_parameter_limits': true,       // 当模型学会参数限制时移除
  'require_manual_approval_l3': true,     // 随信任度提升逐步降低
}
```

每一个硬编码的工作流、决策树和诊断程序都应设定过期日期，并针对更新的模型能力重新测试。
